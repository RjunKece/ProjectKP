4.x.4 Activity Diagram — Melihat Dashboard Admin

Nama Use CaseMelihat Dashboard Admin
AktorAdmin (Super Admin)
DeskripsiAdmin melihat halaman dashboard utama yang menampilkan ringkasan statistik sistem secara keseluruhan, termasuk jumlah pengguna, total aktivitas, grafik tren, distribusi divisi, aktivitas terbaru, dan performa divisi.
Pre-conditionAdmin telah berhasil login. Middleware memverifikasi role = "super_admin".
Post-conditionDashboard admin ditampilkan dengan data statistik real-time dari database.

Alur Utama (Main Flow):

  1. Admin mengklik menu "Dashboard" pada sidebar navigasi atau otomatis diarahkan setelah login.
  2. Sistem menerima request GET ke route /admin/dashboard dan menjalankan AdminDashboardController@index.
  3. Sistem mengirimkan beberapa query ke Database:
    • Query total pengguna: SELECT COUNT(*) FROM users
    • Query pengguna aktif: SELECT COUNT(*) FROM users WHERE role_id = (karyawan)
    • Query total aktivitas: SELECT COUNT(*) FROM activities
    • Query aktivitas hari ini: SELECT COUNT(*) FROM activities WHERE DATE(tanggal) = CURDATE()
    • Query distribusi per divisi: SELECT divisions.nama_divisi, COUNT(users.id) FROM divisions LEFT JOIN users...
    • Query tren aktivitas 12 bulan: SELECT MONTH(tanggal), COUNT(*) FROM activities GROUP BY MONTH(tanggal)
    • Query aktivitas terbaru: SELECT * FROM activities ORDER BY created_at DESC LIMIT 5
  4. Database mengeksekusi seluruh query dan mengembalikan hasil agregasi ke Sistem.
  5. Sistem memproses data dan merender halaman dashboard (admin.dashboard) yang menampilkan:
    • Welcome Banner: Salam dan tanggal hari ini serta status sistem.
    • 6 KPI Cards: Total Users, Karyawan Aktif, Total Aktivitas, Aktivitas Hari Ini, Produktivitas (%), dan Growth (%).
    • Activity Analytics Chart: Line chart tren aktivitas 12 bulan terakhir menggunakan Chart.js.
    • Distribusi Divisi Chart: Doughnut chart jumlah karyawan per divisi.
    • Aktivitas Terbaru: Daftar 5 log aktivitas terkini dari seluruh pengguna.
    • Performa Divisi: Tabel peringkat aktivitas per divisi bulan berjalan.
    • Quick Actions: Tombol akses cepat ke User Management, Aktivitas, Reports, dan Profil.
  6. Admin melihat seluruh informasi dashboard dan dapat bernavigasi ke fitur lain melalui sidebar atau quick actions.

4.x.5 Activity Diagram — Melihat Daftar Karyawan

Nama Use CaseMelihat Daftar Karyawan
AktorAdmin (Super Admin)
DeskripsiAdmin melihat daftar seluruh pengguna yang terdaftar dalam sistem beserta informasi role, divisi, status, dan jumlah aktivitas. Admin dapat melakukan filter berdasarkan nama, role, dan divisi.
Pre-conditionAdmin telah login dan memiliki hak akses super_admin.
Post-conditionDaftar pengguna ditampilkan sesuai dengan filter yang diterapkan.

Alur Utama (Main Flow):

  1. Admin mengklik menu "Users" pada sidebar navigasi.
  2. Sistem menerima request GET ke route /admin/users dan menjalankan UserController@index.
  3. Sistem mengirimkan query ke Database: SELECT users.*, roles.nama_role, divisions.nama_divisi, COUNT(activities.id) as total_activities FROM users LEFT JOIN roles... LEFT JOIN divisions... LEFT JOIN activities... GROUP BY users.id.
  4. Database mengeksekusi query dan mengembalikan data seluruh pengguna beserta relasinya.
  5. Sistem merender halaman User Management (admin.users.index) yang menampilkan:
    • 4 KPI Cards: Total Pengguna, Pengguna Aktif, Super Admin, dan Karyawan.
    • Filter Bar: Input pencarian nama/email, dropdown role, dan dropdown divisi.
    • Tabel Pengguna: Kolom User (avatar, nama, email), Role, Divisi, Status, Jumlah Aktivitas, dan tombol aksi "Reset Password".
  6. Admin melihat daftar seluruh pengguna sistem.

Alur Alternatif — Filter Data:

  1. Admin Melakukan Filter:
    1. Admin mengisi salah satu atau beberapa field filter (nama/email, role, divisi).
    2. Admin mengklik tombol "Filter".
    3. Sistem mengirimkan query dengan klausa WHERE yang sesuai parameter filter ke Database.
    4. Database mengembalikan hasil filter.
    5. Sistem merender ulang tabel dengan data hasil filter.
    6. Admin melihat hasil pencarian/filter.

4.x.6 Activity Diagram — Menambah Pengguna Baru

Nama Use CaseMenambah Data Pengguna / Karyawan
AktorAdmin (Super Admin)
DeskripsiAdmin menambahkan akun pengguna baru ke dalam sistem ERP dengan mengisi data nama, email, password, role, dan divisi melalui form modal pada halaman User Management.
Pre-conditionAdmin berada di halaman User Management. Email yang akan didaftarkan belum digunakan oleh pengguna lain.
Post-conditionData pengguna baru berhasil tersimpan di tabel users. Daftar pengguna diperbarui secara otomatis.

Alur Utama (Main Flow):

  1. Admin mengklik tombol "+ Tambah User" pada halaman User Management.
  2. Sistem menampilkan modal form "Tambah User Baru" yang berisi field: Nama Lengkap, Email, Password, Role (dropdown), dan Divisi (dropdown).
  3. Admin mengisi seluruh field yang tersedia:
    • Nama Lengkap: Nama pengguna baru (wajib).
    • Email: Alamat email unik (wajib, format email valid).
    • Password: Kata sandi minimal 8 karakter (wajib).
    • Role: Memilih dari dropdown — "Super Admin" atau "Karyawan" (wajib).
    • Divisi: Memilih dari dropdown daftar divisi yang tersedia (opsional).
  4. Admin mengklik tombol "Simpan".
  5. Sistem menerima request POST ke route /admin/users dan menjalankan UserController@store.
  6. Sistem melakukan validasi server-side:
    • Nama tidak boleh kosong.
    • Email harus unik dan berformat valid.
    • Password minimal 8 karakter.
    • Role harus berupa ID yang valid dari tabel roles.
  7. Sistem mengenkripsi password menggunakan algoritma Bcrypt.
  8. Sistem mengirimkan query INSERT ke Database: INSERT INTO users (name, email, password, role_id, division_id) VALUES (...).
  9. Database menyimpan record baru dan mengembalikan konfirmasi ke Sistem.
  10. Sistem menutup modal form dan menampilkan notifikasi sukses: "User berhasil ditambahkan!"
  11. Sistem merefresh daftar pengguna pada tabel untuk menampilkan data terbaru.
  12. Admin melihat data pengguna baru pada tabel dan notifikasi sukses.

Alur Alternatif (Alternative Flow):

  1. Validasi Gagal (Email Duplikat):
    1. Pada langkah 6, apabila email sudah terdaftar dalam database, Sistem mengembalikan pesan error: "Email sudah digunakan oleh pengguna lain."
    2. Modal form tetap terbuka dengan data yang telah diisi.
    3. Admin memperbaiki email dan mengulangi langkah 4.
  2. Admin Membatalkan:
    1. Admin mengklik tombol "Batal" pada modal form.
    2. Sistem menutup modal tanpa menyimpan data apapun.
    3. Alur selesai, kembali ke tampilan tabel pengguna.

4.x.7 Activity Diagram — Reset Password Akun Karyawan

Nama Use CaseMelakukan Reset Password Akun Karyawan
AktorAdmin (Super Admin)
DeskripsiAdmin melakukan reset password untuk akun karyawan yang lupa password atau memerlukan pengaturan ulang akses. Proses ini dilakukan melalui tombol "Reset Password" pada tabel User Management.
Pre-conditionAdmin berada di halaman User Management. Terdapat minimal satu pengguna yang terdaftar.
Post-conditionPassword akun karyawan berhasil diubah di database. Karyawan dapat login dengan password baru.

Alur Utama (Main Flow):

  1. Admin mengidentifikasi akun karyawan yang perlu di-reset pada tabel daftar pengguna.
  2. Admin mengklik tombol "🔑 Reset Password" pada baris pengguna yang dipilih.
  3. Sistem menampilkan modal konfirmasi yang berisi informasi pengguna target dan field input "Password Baru".
  4. Admin mengisi field password baru (minimal 8 karakter).
  5. Admin mengklik tombol "Reset" untuk mengkonfirmasi perubahan.
  6. Sistem menerima request PUT ke route /admin/users/{user}/reset-password dan menjalankan UserController@resetPassword.
  7. Sistem memvalidasi input password baru (minimal 8 karakter).
  8. Sistem mengenkripsi password baru menggunakan algoritma Bcrypt.
  9. Sistem mengirimkan query UPDATE ke Database: UPDATE users SET password = ? WHERE id = ?.
  10. Database memperbarui record password dan mengembalikan konfirmasi.
  11. Sistem menutup modal dan menampilkan notifikasi sukses: "Password berhasil di-reset!"
  12. Admin melihat notifikasi dan karyawan dapat login dengan password baru.

Alur Alternatif (Alternative Flow):

  1. Password Terlalu Pendek:
    1. Pada langkah 7, apabila password baru kurang dari 8 karakter, Sistem menampilkan pesan error validasi.
    2. Modal tetap terbuka. Admin memperbaiki input dan mengulangi langkah 5.

4.x.8 Activity Diagram — Melihat Rekam Jejak Aktivitas Karyawan

Nama Use CaseMelihat Rekam Jejak Aktivitas Karyawan
AktorAdmin (Super Admin)
DeskripsiAdmin memantau seluruh rekam jejak aktivitas yang dicatat oleh semua karyawan dalam sistem. Halaman ini menampilkan statistik global, grafik tren, breakdown status, dan tabel aktivitas dengan fitur filter dan pagination.
Pre-conditionAdmin telah login. Terdapat data aktivitas dalam database.
Post-conditionData aktivitas seluruh pengguna berhasil ditampilkan sesuai filter yang diterapkan.

Alur Utama (Main Flow):

  1. Admin mengklik menu "Aktivitas" pada sidebar navigasi.
  2. Sistem menerima request GET ke route /admin/activities dan menjalankan ActivityController@index.
  3. Sistem mengirimkan query ke Database untuk mengambil data aktivitas seluruh pengguna:
    • SELECT activities.*, users.name, divisions.nama_divisi FROM activities JOIN users... JOIN divisions...
    • Query agregasi untuk KPI: total aktivitas, aktivitas hari ini, aktivitas 7 hari, breakdown per role.
    • Query tren mingguan: SELECT DATE(tanggal), COUNT(*) FROM activities WHERE tanggal >= (7 hari lalu) GROUP BY DATE(tanggal)
  4. Database mengeksekusi seluruh query dan mengembalikan hasil ke Sistem.
  5. Sistem merender halaman Monitoring Aktivitas (admin.activities.index) yang menampilkan:
    • 5 KPI Cards: Total Aktivitas, Hari Ini, 7 Hari Terakhir, Aktivitas Admin, dan Aktivitas Karyawan.
    • Chart Tren 7 Hari: Bar chart menggunakan Chart.js.
    • Status Breakdown: Progress bar untuk setiap status (Submitted, Completed, Approved, Cancelled).
    • Filter Bar: Input pencarian, dropdown role, divisi, status, dan date picker.
    • Tabel Aktivitas: Kolom nomor, User (avatar + nama + divisi), Deskripsi, Status, dan Waktu dengan pagination.
  6. Admin melihat data monitoring dan dapat melakukan filter untuk mempersempit pencarian.

Alur Alternatif — Filter Data:

  1. Admin Menggunakan Filter:
    1. Admin mengisi parameter filter (pencarian nama, role, divisi, status, atau tanggal).
    2. Admin mengklik tombol "Filter".
    3. Sistem menambahkan klausa WHERE pada query berdasarkan parameter yang diisi.
    4. Database mengembalikan data yang sesuai dengan filter.
    5. Sistem merender ulang tabel dengan data hasil filter dan pagination yang diperbarui.
  2. Admin Mereset Filter:
    1. Admin mengklik tombol "Reset".
    2. Sistem menghapus seluruh parameter filter dan menampilkan kembali data lengkap.

4.x.9 Activity Diagram — Mengelola Laporan

Nama Use CaseMengelola Laporan
AktorAdmin (Super Admin)
DeskripsiAdmin mengelola seluruh laporan perusahaan melalui Reports Center. Fitur ini mencakup tiga sub-fungsi utama: (1) Men-generate laporan baru, (2) Menyimpan laporan ke database, dan (3) Melihat arsip keseluruhan laporan. Halaman juga menyediakan Report Builder dengan 4 tipe laporan standar.
Pre-conditionAdmin telah login dengan role super_admin.
Post-conditionLaporan berhasil dibuat/disimpan/dilihat sesuai aksi yang dipilih.

Alur Utama (Main Flow) — Generate Laporan Baru:

  1. Admin mengklik menu "Reports" pada sidebar navigasi.
  2. Sistem menerima request dan menjalankan ReportController@index.
  3. Sistem mengirimkan query ke Database untuk mengambil data laporan dan KPI.
  4. Database mengembalikan data dan Sistem merender halaman Reports Center dengan:
    • 5 KPI Cards: Total Reports, Bulan Ini, Laporan Perusahaan, Laporan Divisi, dan Pending.
    • Report Builder: 4 tipe laporan — Financial Intelligence, Department Performance, Growth & Trends, dan Custom Intelligence.
    • Tabel Arsip: Seluruh laporan yang pernah dibuat.
  5. Admin memilih tipe laporan pada Report Builder, misalnya "Financial Intelligence".
  6. Admin mengklik tombol "Generate" pada tipe yang dipilih.
  7. Sistem menampilkan modal konfigurasi yang berisi field:
    • Judul Laporan: Judul deskriptif (wajib).
    • Scope: Perusahaan atau Divisi tertentu (wajib).
    • Periode: Tanggal awal dan tanggal akhir (opsional).
    • Prioritas: Low, Normal, High, Urgent (default: Normal).
    • Deskripsi: Isi atau ringkasan laporan (opsional).
  8. Admin mengisi seluruh field yang diperlukan.
  9. Admin mengklik tombol "Simpan & Generate".
  10. Sistem menerima request POST ke /admin/reports/store dan menjalankan ReportController@store.
  11. Sistem melakukan validasi data (judul wajib, tipe valid, scope valid).
  12. Sistem mengirimkan query INSERT ke Database: INSERT INTO reports (title, type, scope, division_id, description, start_date, end_date, status, priority, created_by) VALUES (...).
  13. Database menyimpan record laporan baru dengan status "generated".
  14. Sistem menutup modal dan menampilkan notifikasi: "Laporan berhasil dibuat!".
  15. Sistem merefresh halaman, dan laporan baru muncul pada tabel arsip.

Alur Alternatif — Melihat Arsip Laporan:

  1. Admin Melihat Detail Laporan:
    1. Admin mengklik tombol "👁 View" pada baris laporan di tabel arsip.
    2. Sistem mengirimkan request GET ke /admin/reports/{id} dan query detail ke Database.
    3. Database mengembalikan data laporan beserta daftar tanggapan (report_responses).
    4. Sistem menampilkan modal detail yang berisi: header laporan, isi lengkap, metadata, dan daftar respon.
    5. Admin membaca isi laporan.
    6. Admin menutup modal atau mengklik tombol "Print" untuk mencetak.
  2. Admin Menghapus Laporan:
    1. Admin mengklik tombol "🗑 Hapus" pada baris laporan.
    2. Sistem menampilkan dialog konfirmasi: "Yakin ingin menghapus laporan ini?"
    3. Admin mengkonfirmasi penghapusan.
    4. Sistem mengirimkan request DELETE ke /admin/reports/{id}.
    5. Database menghapus record dari tabel reports (cascade delete ke report_responses).
    6. Sistem merefresh tabel dan menampilkan notifikasi: "Laporan berhasil dihapus!".
