Dokumentasi Activity Diagram
Sistem ERP PT. Golden Intan Berlian

Berikut ini adalah dokumentasi lengkap activity diagram dari seluruh fitur dan menu yang tersedia dalam Sistem ERP PT. Golden Intan Berlian. Setiap activity diagram menggambarkan alur kerja (workflow) interaksi antara aktor (Pengguna), Sistem, dan Database sesuai dengan standar notasi UML (Unified Modeling Language) 2.0. Dokumentasi ini disusun sebagai bagian dari Bab 3 (Perancangan Sistem) dan Bab 4 (Hasil dan Pembahasan) laporan Kerja Praktik.

Sistem ini memiliki dua aktor utama, yaitu Admin (Super Admin) yang bertanggung jawab atas pengelolaan sistem secara keseluruhan, dan Karyawan yang menggunakan sistem untuk keperluan operasional harian. Setiap activity diagram menggunakan tiga swimlane: Pengguna (Aktor), Sistem (Aplikasi Laravel), dan Database (MySQL/MariaDB).

4.x.1 Activity Diagram — Proses Login

Nama Use CaseMelakukan Login
AktorAdmin (Super Admin) dan Karyawan
DeskripsiProses autentikasi pengguna untuk masuk ke dalam sistem ERP menggunakan email dan password. Sistem akan melakukan validasi kredensial terhadap database dan mengarahkan pengguna ke dashboard sesuai role masing-masing.
Pre-conditionPengguna telah memiliki akun yang terdaftar dalam sistem (dibuat oleh Admin). Pengguna belum dalam keadaan login (session tidak aktif).
Post-conditionPengguna berhasil masuk ke dalam sistem dan diarahkan ke halaman dashboard sesuai role (Admin Dashboard atau Karyawan Dashboard). Session pengguna tercatat aktif.

Alur Utama (Main Flow):

  1. Pengguna membuka halaman login sistem ERP melalui browser dengan mengakses URL /login.
  2. Sistem menampilkan halaman login yang terdiri dari form input email, password, dan tombol "Sign In".
  3. Pengguna mengisi field Email Address dengan alamat email yang terdaftar.
  4. Pengguna mengisi field Password dengan kata sandi akun.
  5. Pengguna menekan tombol "Sign In" untuk mengirim request autentikasi.
  6. Sistem menerima input data (email dan password) dari form login.
  7. Sistem mengirimkan request validasi ke Database untuk memverifikasi kredensial.
  8. Database menerima query dan melakukan pencocokan data pada tabel users berdasarkan email dan password (Bcrypt hash comparison).
  9. Database mengembalikan hasil validasi ke Sistem.
  10. Sistem memeriksa hasil validasi — Data Valid.
  11. Sistem melakukan pengecekan role pengguna melalui relasi users.role_id → tabel roles.
  12. Sistem menentukan tujuan redirect berdasarkan nilai nama_role:
    • Jika nama_role = "super_admin" → Sistem menyiapkan dan melakukan redirect ke halaman Admin Dashboard (/admin/dashboard).
    • Jika nama_role = "karyawan" → Sistem menyiapkan dan melakukan redirect ke halaman Karyawan Dashboard (/karyawan/dashboard).
  13. Sistem membuat session baru untuk pengguna dan menampilkan halaman dashboard yang sesuai.
  14. Pengguna melihat halaman dashboard dan dapat mulai menggunakan fitur-fitur sistem sesuai hak aksesnya.

Alur Alternatif (Alternative Flow):

  1. Validasi Gagal (Data Tidak Valid):
    1. Pada langkah 10, apabila Database tidak menemukan kecocokan data (email tidak terdaftar atau password salah), Database mengembalikan hasil "tidak valid".
    2. Sistem menampilkan pesan error: "Email atau password yang Anda masukkan salah. Silakan coba lagi."
    3. Pengguna melihat pesan error pada halaman login.
    4. Alur kembali ke langkah 3 — pengguna dapat mengisi ulang kredensial.
  2. Field Kosong (Validasi Form):
    1. Pada langkah 5, apabila pengguna menekan tombol "Sign In" tanpa mengisi salah satu atau kedua field, Sistem melakukan validasi client-side.
    2. Sistem menampilkan pesan validasi: "Field email dan password wajib diisi."
    3. Alur kembali ke langkah 3.

4.x.2 Activity Diagram — Melihat Profil

Nama Use CaseMelihat Profil
AktorAdmin (Super Admin) dan Karyawan
DeskripsiPengguna melihat informasi profil akun pribadi yang mencakup nama, email, role, divisi, tanggal registrasi, dan ringkasan aktivitas. Fitur ini tersedia untuk kedua role dan dapat diakses melalui menu sidebar "Profil".
Pre-conditionPengguna telah berhasil login ke dalam sistem (session aktif).
Post-conditionHalaman profil berhasil ditampilkan dengan data yang sesuai dari database.

Alur Utama (Main Flow):

  1. Pengguna mengklik menu "Profil" pada sidebar navigasi.
  2. Sistem menerima request dan menjalankan method ProfileController@index.
  3. Sistem mengirimkan query ke Database untuk mengambil data user yang sedang login berdasarkan auth()->user()->id.
  4. Database mengeksekusi query pada tabel users dengan join ke tabel roles dan divisions, serta menghitung total aktivitas dari tabel activities.
  5. Database mengembalikan data pengguna beserta relasi dan agregasi ke Sistem.
  6. Sistem memproses data dan merender halaman profil (profile.index) yang menampilkan:
    • Avatar dan Identitas: Nama lengkap, email, badge role, dan status akun.
    • Account Information: Role, divisi, dan tanggal registrasi.
    • Activity Summary: Total aktivitas keseluruhan dan total aktivitas bulan berjalan.
    • Quick Actions: Tombol refresh profil dan tombol logout.
  7. Pengguna melihat seluruh informasi profil pada halaman.

Alur Alternatif (Alternative Flow):

  1. Session Expired:
    1. Pada langkah 2, apabila session pengguna telah berakhir (expired), middleware auth mendeteksi bahwa pengguna tidak terautentikasi.
    2. Sistem melakukan redirect otomatis ke halaman login.
    3. Pengguna harus melakukan login ulang sebelum dapat mengakses halaman profil.

4.x.3 Activity Diagram — Proses Logout

Nama Use CaseLogout
AktorAdmin (Super Admin) dan Karyawan
DeskripsiPengguna keluar dari sistem ERP. Sistem akan menghapus session aktif, melakukan invalidasi token CSRF, dan mengarahkan pengguna kembali ke halaman login.
Pre-conditionPengguna dalam keadaan login (session aktif).
Post-conditionSession pengguna dihapus. Pengguna diarahkan ke halaman login dengan pesan konfirmasi logout berhasil.

Alur Utama (Main Flow):

  1. Pengguna mengklik tombol "Logout" yang tersedia pada sidebar navigasi atau halaman profil.
  2. Sistem menerima request POST ke route /logout.
  3. Sistem melakukan verifikasi token CSRF untuk memastikan keabsahan request.
  4. Sistem menjalankan fungsi Auth::logout() untuk menghapus autentikasi pengguna.
  5. Sistem menjalankan session()->invalidate() untuk menghapus seluruh data session dari server.
  6. Sistem menjalankan session()->regenerateToken() untuk membuat token CSRF baru guna mencegah serangan CSRF.
  7. Database menghapus record session pada tabel sessions yang terkait dengan user_id pengguna.
  8. Sistem melakukan redirect ke halaman login (/login) dengan pesan flash: "Anda telah berhasil logout. Silahkan login kembali untuk melanjutkan."
  9. Pengguna melihat halaman login dengan pesan konfirmasi logout berhasil.

Alur Alternatif (Alternative Flow):

  1. Token CSRF Tidak Valid:
    1. Pada langkah 3, apabila token CSRF tidak valid (misalnya karena tab browser sudah lama terbuka), Sistem menolak request dengan HTTP 419 (Page Expired).
    2. Pengguna perlu merefresh halaman dan mengklik logout kembali.
